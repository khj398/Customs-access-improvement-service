# Demo 구현 실행 계획 v3 (D-1 집중안: 로컬 DB + 프론트 + OpenAI 보강 분류 포함)

## 0) 이번 데모 목표 (하루 안에 반드시 보여줄 것)
이번 1차 데모의 성공 기준은 아래 3가지입니다.

1. 앱에서 물품이 안정적으로 잘 보인다.
2. 검색이 잘 동작한다(사용자 키워드 기준으로 납득 가능한 결과).
3. 같이 묶여있는 물품(동일 공매번호/연관 물품)을 자연스럽게 보여준다.

> 서버 정식 배포는 제외하되, **로컬 전용 API**로 프론트 연동을 완성한다.

---

## 1) 핵심 의사결정 (기존 계획 보정)

- 일정 제약: **데모까지 1일**
- 1차 범위: **OpenAI 보강 분류를 반드시 포함**
- 데이터 전략: **최신 공매 데이터 + 과거 데이터 일부 혼합**
  - 이유: 현재 사이트 실물품 수가 적어 데모 다양성이 부족함

즉, "rule만"이 아니라 **rule 1차 + OpenAI 2차 보강**까지 오늘 안에 데모 가능한 수준으로 넣는다.

---

## 2) 데이터 구성 전략 (최신 + 과거 혼합)

### 2-1. 데이터 소스
1. 최신 데이터(필수)
   - `project/AWSLambda/UNIPASS_LIST_Business.py`
   - `project/AWSLambda/UNIPASS_LIST_Personal.py`
2. 과거 데이터(보강)
   - 기존 JSON 백업본에서 데모용 샘플 선별
3. 이미지(가능하면)
   - `project/AWSLambda/UNIPASS_Image.py`

### 2-2. 혼합 원칙
- 최신 데이터는 "실시간성"을 보여주는 용도
- 과거 데이터는 "검색/카테고리 다양성"을 확보하는 용도
- 데모 DB에는 `data_source` 성격(예: `live`, `archive`)이 구분되도록 메모 또는 매핑 테이블/필드로 표시

### 2-3. 최소 목표 수량 (데모 안정 기준)
- 전체 노출 물품: 80~150개
- 카테고리: 최소 6개 이상
- 검색 테스트 키워드: 최소 10개(한글/영문 혼합)

---

## 3) 파이프라인 (오늘 실행 순서)

1. MySQL 스키마/시드 반영
2. 최신 수집 실행
3. 과거 샘플 JSON 병합(데모용)
4. ETL 적재
   - `etl/load_unipass_to_mysql.py`
5. Rule 분류/토큰 생성
   - `classification/build_classification.py`
6. OpenAI 보강 분류 실행(필수)
   - 대상: 미분류 + 저신뢰 + 검색 중요 품목
   - 결과 반영: `item_classification`에 `model_name`, `model_ver`, `confidence`, `rationale` 기록

---

## 4) 로컬 API 범위 (프론트 연결용, 읽기 전용)

### 4-1. 필수 엔드포인트
- `GET /health`
- `GET /items?query=&category=&limit=&offset=&sort=`
- `GET /items/{item_id}`
- `GET /categories`
- `GET /items/{item_id}/grouped`  ← 같이 묶인 물품 조회
- `GET /search/suggest?q=`        ← 검색 추천어(선택이지만 강력 권장)

### 4-2. 같이 묶인 물품 정의
우선순위:
1. 동일 `pbac_no` (같은 공매번호)
2. 동일/인접 카테고리
3. 토큰 유사도(동의어/핵심키워드)

1차 데모에서는 **동일 `pbac_no` 기준만으로도 충분히 설득력 있는 "묶음" 화면 구현 가능**.

---

## 5) 검색 기능 구현 전략 (1일 데모 최적화)

### 5-1. 검색 품질 우선순위
1. 정확 일치: 물품명/동의어 직접 매칭
2. 부분 일치: prefix/contains
3. 보강 일치: 분류 카테고리/토큰 기반 매칭

### 5-2. 최소 구현 포인트
- `item_search_token`을 적극 활용해 한글/영문 검색 동작 보장
- query 정규화(소문자, 특수문자 제거, 공백 정리)
- 정렬 가중치(예시)
  - 정확 일치 > 토큰 일치 > 카테고리 일치 > 최신순

### 5-3. 데모용 검색 시나리오 세트
- "와인", "battery", "차량 부품", "전자", "주류", "냉장고"
- 검색 0건일 때는 유사 카테고리/추천 검색어 노출

---

## 6) 프론트 데모 화면 구성 (필수 3화면)

1. 목록 화면
   - 검색창 + 카테고리 필터 + 정렬 + 카드 목록
2. 상세 화면
   - 이미지, 물품 정보, 분류 정보(OpenAI 보강 여부 표시)
3. 묶음/연관 화면(또는 상세 하단 섹션)
   - "같은 공매 묶음" 물품 리스트

> 첫 데모는 디자인 완성도보다 **데이터 신뢰 + 탐색 경험**을 우선한다.

---

## 7) D-1 실행 타임라인 (현실형)

### 오전 (3~4시간)
- 최신 수집 + 과거 샘플 병합 + ETL 완료
- rule 분류 실행

### 오후 초반 (2~3시간)
- OpenAI 보강 분류 실행 및 DB 반영
- 검색 핵심 쿼리 점검

### 오후 후반 (2~3시간)
- 로컬 API 연결 및 프론트 목록/상세/묶음 완성
- 검색 시나리오 10개 리허설

### 마감 전 (1시간)
- 데모 스크립트 고정(발표 순서/키워드/백업 화면)
- 실패 대응 문구 준비(검색 0건, 이미지 없음, API 지연)

## 8) 리스크와 즉시 대응

### 리스크 A. OpenAI 분류 시간이 지연됨
- 대응: 전체가 아닌 우선순위 품목만 보강 분류(상위 100건)

### 리스크 B. 최신 데이터 수량이 너무 적음
- 대응: 과거 데이터 샘플 비중 확대(단, 최신 탭/뱃지 분리)

### 리스크 C. 검색 결과 신뢰도가 낮음
- 대응: 토큰 사전(동의어) 즉시 보강 + 쿼리 가중치 조정

### 리스크 D. 묶음 물품이 빈약해 보임
- 대응: `pbac_no` 단위 묶음이 풍부한 샘플 우선 노출

## 9) 오늘 합의할 토론 안건
1. OpenAI 보강 분류 대상 범위(전체 vs 상위 N건)

3. "묶음 물품" 정의를 1차에서 `pbac_no`로 고정할지
4. 검색 추천어(`GET /search/suggest`)를 1차 필수로 넣을지
